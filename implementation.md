# Contextualize Debug + Issue Memory Implementation

## What was implemented

Two new CLI workflows were added in `bin/contextualize.js`:

- `contextualize debug`
- `contextualize add issue`

The implementation is designed around your requested pipeline:

1. Stack trace intake (`--trace`, last saved trace, or interactive paste)
2. Concurrent context retrieval (similar issues + doc context)
3. Dependency-issue detection signal
4. OpenAI `gpt-4.1-mini` RCA + fix suggestion
5. Terminal output + generated markdown prompt for agent handoff

## New command behavior

### `contextualize add issue`

Interactive, multi-turn CLI flow:

- Prompts for:
  - issue title
  - labels (comma separated)
  - issue details / resolution notes
- Shows suggested labels:
  - `bug`, `dependency`, `config`, `api`, `llm`, `performance`, `enhancement`, `infra`
- Auto-generates backend ID (`issue_<timestamp>`)
- Saves issue locally to:
  - `.contextualize/issues_store.json`
- Then asks whether to also upsert to Vertex AI index

Vertex upsert path:

- Reads `.contextualize/vertex_config.json`
- Uses Python + Vertex SDK (`vertexai` + `google.cloud.aiplatform`)
- Embeds issue text with `text-embedding-005`
- Calls `MatchingEngineIndex.upsert_datapoints(...)`

If Vertex config is incomplete/missing, local storage still works and the CLI prints a clear message.

### `contextualize debug`

Debug pipeline behavior:

1. **Get trace**
   - `--trace "..."` if provided
   - else `.contextualize/last_error.log` if available
   - else interactive prompt
2. **Persist trace**
   - writes latest trace back to `.contextualize/last_error.log`
3. **Concurrent retrieval**
   - Similar issues:
     - tries Vertex nearest-neighbor query first (if endpoint config exists)
     - falls back to local issue store matching
   - Documentation RAG:
     - token overlap scoring over `.contextualize/docs/*.md`
     - picks top matching snippets
4. **Dependency signal**
   - heuristics detect module/package missing patterns
   - cross-checks `.contextualize/scan/dependencies.json`
5. **Intelligence step**
   - calls OpenAI `gpt-4.1-mini`
   - prompt includes: trace + dependency signal + retrieved issues + docs snippets
   - streams output to terminal
6. **Prompt artifact**
   - writes `.contextualize/debug/fix_prompt.md`
   - optional HITL step to invoke Codex:
     - `codex "$(cat .contextualize/debug/fix_prompt.md)"`

## File/folder structure used

No major file moves were made. The command uses and/or creates:

- `bin/contextualize.js` (updated)
- `.contextualize/last_error.log` (new lifecycle file)
- `.contextualize/issues_store.json` (new local issue memory)
- `.contextualize/debug/fix_prompt.md` (generated by debug)
- `.contextualize/vertex_config.json` (existing expected config, reused)

`contextualize init` now ensures all required `.contextualize` layout elements exist, including `last_error.log`, `issues_store.json`, and `.contextualize/debug/`.

## Vertex AI assumptions and requirements

For Vertex retrieval/upsert to work end-to-end:

- Credentials are configured for Google Cloud auth
- Python dependencies are installed:
  - `google-cloud-aiplatform`
  - `vertexai` package support
- `.contextualize/vertex_config.json` contains:
  - `project_id`
  - `region` (optional, defaults to `us-central1`)
  - `index_name` (for upsert)
  - `index_endpoint_name` + `deployed_index_id` (for nearest-neighbor query during debug)

If endpoint fields are missing, debug gracefully falls back to local issue memory retrieval.

## Notes on design

- The implementation keeps CLI experience interactive and demo-friendly.
- Local issue memory enables immediate usage even without many GitHub issues.
- Vertex integration is opportunistic: used when configured, non-blocking when not.
- The generated markdown prompt preserves all retrieved context for downstream coding agents.
